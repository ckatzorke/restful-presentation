What is REST

  • REpresentational State Transfer
  • Architectural style
    • Stateless
    • Client-server
    • Cacheable
    • Layered
    • Uniform Interface (resource identification and manipulation, self-descriptive, HATEOAS)
    • Code on demand (optional)
  • HTTP is a RESTFul implementation

Richardson Maturity Model

Level

  • 0 using HTTP for transport, RPC over HTTP
  • 1 using resources
  • 2 HTTP verbs (and status codes)
  • 3 Hypermedia controls

Resources

  • Can be an object or just an abstract concept
  • Must be uniquely addressable
  • Can be a container of other resources
  • Canonical resource URI

Examples

  • https://server.com/users (all users)
  • https://server.com/users/89562912 (single user)
  • https://server.com/users/89562912/orders (nested resource, all orders)
  • ...

Uniform Interface - Media types / content negotiation

Accept and Content-Type HTTP headers can be used to describe format for retrieving and sending the content.

Alternatively, use query parameter (?format=json) or URI extensions (resource.xml)

Uniform Interface - HTTP verbs

Use HTTP verbs to query and manipulate resources. Use them in the way they were meant to be used!

  • GET
  • PUT
  • POST *
  • DELETE
  • HEAD
  • OPTIONS
  • PATCH

* Not idempotent

PUT vs POST

Use PUT with sending the full content to create a resource

Use POST to create or update subordinates of a resource.

Use Location header in response to send the new resource URI.

PATCH

Designed for partial updates, list of changes provided in form of a PATCH document (e.g. application/json-patch+json).

Uniform Interface - HTTP status codes

Make proper use of HTTP status codes. They are the best way for describing behaviour and responses.

  • 200 OK
  • 201 created
  • 202 accepted
  • 302 found (but temporarily somewhere else)
  • 304 not modified
  • 404 not found
  • 409 conflict

But keep it simple. No need to use all of them, no need to invent new ones.

Hypermedia as the Engine of the Application State

"A hypermedia-driven site provides information to navigate the site's REST interfaces dynamically by including hypermedia links with the responses."

https://spring.io/understanding/HATEOAS

Comparable to a homepage of a website.


							{
								"name": "Alice",
								"links": [ {
									"rel": "self",
									"href": "http://localhost:8080/customer/1"
								} ]
							}

Cacheability

Use caching for representations.

Use Last-Modified and ETag in responses and If-Modified-Since and If-None-Match in requests.
A server can send a 304 (not modified), so you can reuse your current state representation.

Versioning

Version your api and your representation.

Use URI part or Accept content types (e.g. application/vnd.company.myapp.customer-v3+json).

A version is out-of-band information, and must not be part of URL (according to Fielding), but are more convenient to use. Use hypermedia to mitigate this.

Common REST mistakes

  • Using RPC over HTTP
  • Overusing single verbs (esp. GET and POST)
  • Actions in URI (/readuser /deletecustomer)
  • Creating state on the server
  • Ignoring status codes
  • Ignoring caching

						POST /jobs
						HTTP/1.1 202 Accepted
						Location: /queue/123 

						GET /queue/123
						HTTP/1.1 200 OK
						{
